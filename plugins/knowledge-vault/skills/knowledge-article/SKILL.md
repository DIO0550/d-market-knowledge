# Knowledge Article

`dev-knowledge` リポジトリの蓄積ルールに沿った、ナレッジ記事の Markdown を 1 本作成する。

---

## いつ使うか

- 開発中に遭遇した問題と解決策を記録に残したいとき
- 得た知見・設計判断を後から検索・再利用できる形で蓄積したいとき
- このスキルで記事を作ったあと、`knowledge-issue` で GitHub Issue 化する流れが基本

## dev-knowledge の蓄積ルール（前提）

- **雑多に蓄積を優先**。整理コストで蓄積のハードルを上げない。
- **1 記事 = 1 つの問題 or 知見**。粒度は小さく。複数トピックを 1 記事に詰め込まない。
- 横断検索は **tags が主役**。技術名・トピック名を惜しまず付ける。
- 記事・コメントは **日本語**で書く。
- 過剰な階層化はしない。カテゴリ（フォルダ）は粗く保つ。

## frontmatter

各 `.md` の冒頭に必ず付ける。

```yaml
---
title: 簡潔なタイトル
tags: [react, hooks, ...]
---
```

- `title`: 内容が一目で分かる簡潔な日本語。
- `tags`: 技術名・トピック名を複数。横断検索の起点になるので惜しまない。

## 記事本文の基本構成

「遭遇した問題 → 原因 → 解決」を基本の流れにする。再現条件と環境を残す。

```markdown
---
title: <簡潔なタイトル>
tags: [<tech>, <topic>, ...]
---

## TL;DR

- 結論を箇条書きで先に。

## 遭遇した問題

- 何が起きたか。再現条件・環境（バージョン等）を残す。

## 原因

- なぜそうなるか。

## 解決

- どう直したか / どう設計したか。

\`\`\`ts
// 最小再現に絞ったコード。動かない例と動く例を併記すると後で役立つ。
\`\`\`

## まとめ / 参考

- 参考リンクや一行まとめ。
```

- 知見系（問題ではなく設計指針など）の場合は「問題」を「このドキュメントの射程」などに置き換えてよい。流れの本質は「何を / なぜ / どうする」。
- コードは**最小再現に絞る**。長いコードを貼らない。

## 保存先

技術ごとにフォルダを分ける。

```
docs/
  <tech>/                  # react, rust, claude-code, ...（無ければ新規作成）
    <簡潔なタイトル>.md
```

- 該当する技術フォルダが無ければ新規作成してよい（カテゴリは粗くてよい）。
- ファイル名は記事タイトルに対応した日本語で問題ない（既存記事もそうなっている）。
- Docusaurus 構成のため、実体は `dev-knowledge/dev-knowledge/docs/<tech>/` 配下にある点に注意（リポジトリ直下ではなくネストした `dev-knowledge/` の中）。

## ワークフロー

1. 記録したい「1 つの問題 or 知見」を 1 つに絞る。複数あれば記事を分ける。
2. `title` と `tags` を決める（tags は多めに）。
3. 「何が起きたか → なぜか → どうしたか」で本文を書く。再現条件・環境・最小コードを残す。
4. 保存先の技術フォルダを決め、`.md` を作成する。
5. 必要なら `knowledge-issue` で GitHub Issue として登録する。

## 注意点

- 1 記事に複数トピックを詰め込まない。迷ったら分割する。
- tags を省略しない。あとで自分が検索する唯一の手がかりになる。
- 整理しすぎて書くのをやめるくらいなら、粗くてもまず残す。
