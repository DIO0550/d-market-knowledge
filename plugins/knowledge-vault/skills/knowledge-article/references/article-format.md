# ナレッジ記事フォーマット（dev-knowledge 準拠）

`dev-knowledge` リポジトリの既存 md を参考にした、記事ファイルの型。

## Contents

- frontmatter
- 本文の型

- 守るルール

---

## frontmatter（必須）

```yaml
---
title: 簡潔なタイトル
tags: [react, hooks, ...]
---
```

- `title`: 内容が一目で分かる簡潔な日本語。
- `tags`: 技術名・トピック名を複数。横断検索の起点なので惜しまない。

## 本文の型

「遭遇した問題 → 原因 → 解決」を基本の流れにする。再現条件と環境を残す。
問題ではなく設計指針などの知見系は「遭遇した問題」を「このドキュメントの射程」等に置き換えてよい。型の本質は「何を / なぜ / どうする」。

## 完成テンプレート

## TL;DR

- 何が起きたか。再現条件・環境（バージョン等）を残す。

## 原因

- なぜそうなるか。

## 解決

- どう直したか / どう設計したか。

```ts
// 最小再現に絞ったコード。動かない例と動く例を併記すると後で役立つ。
```

## まとめ / 参考

- 参考リンクや一行まとめ。
```

## 守るルール（dev-knowledge の方針）

- **1 記事 = 1 トピック**。複数を 1 ファイルに詰め込まない。迷ったら分割。
- **tags は省略しない**。あとで検索する唯一の手がかり。
- 記事・コメントは**日本語**で書く。
- コードは**最小再現に絞る**。長いコードは貼らない。
- 整理しすぎて書くのをやめるより、粗くてもまず残す。
