# 会話からナレッジ候補を棚卸しして issue にまとめて起票する

会話の中で「これは後で再利用できる」技術的な知見（言語・フレームワーク・API の挙動、設計手法、ハマりどころなど）は、その場で記録しないと流れて消える。**狙いは技術ナレッジの収集であって、使っていたスキルやツールの使い方の記録ではない。**このスキルは、セッションを振り返って **ナレッジ候補を拾い上げ → ユーザーが選別 → まとめて issue 化** する受け口。

`knowledge-article`（1記事を書く）/ `knowledge-issue`（1件を起票する）に対し、このスキルは **複数候補を一度に洗い出して仕分ける** 役を担う。1件ずつの起票そのものは `knowledge-issue` のルールに従う。

設計の柱は3つ：**①対象の会話はユーザーが選ぶ（`AskUserQuestion`）②候補の情報収集はサブエージェントに任せる ③どれを起票するかはユーザーが決める（`AskUserQuestion`）**。Claude が勝手に対象を決めたり全件起票したりしない。既存 issue との重複は気にしない（被ってよい。目的は知見を集めること）。

---

## ワークフロー

進捗チェックリストをコピーして進める：

```
- [ ] Step 1: 対象の会話を AskUserQuestion でユーザーに選ばせる
- [ ] Step 2: サブエージェントでナレッジ候補を情報収集する
- [ ] Step 3: 候補リストを提示し、どれを起票するか AskUserQuestion でユーザーに決めさせる
- [ ] Step 4: 選ばれた候補を 1 件ずつ起票する
- [ ] Step 5: 起票結果を報告する
```

> このスキルはユーザーの判断が要る場面が2回ある（対象の会話／起票する候補）。**どちらも原則 `AskUserQuestion` ツールを使って明示的に選ばせる。** Claude が勝手に決めたり、曖昧な聞き方で進めたりしない。
>
> **`AskUserQuestion` が使えない環境でのフォールバック（重要）**: 実行環境によっては `AskUserQuestion` 相当のツールが無い（例: Codex など他の CLI、非対話実行）。その場合は **番号付きの平文プロンプト** で代替する — 選択肢を `1.` `2.` … と列挙し、「番号で答えてください（複数可）」と明示して**ユーザーの返答を待つ**。ここで禁止なのは「曖昧な聞き方」と「Claude の独断」であって、構造化した平文での選択依頼は OK。**候補だけ出して止まる／黙って既定で進める／全件を勝手に起票する、のいずれもしない。** どの環境でも「ユーザーに明示的に選ばせる」という設計の柱は不変。

### Step 1: 対象の会話を AskUserQuestion でユーザーに選ばせる

**勝手に既定で進めない。** どの会話を棚卸し対象にするかを **`AskUserQuestion` ツールで**（無い環境では上記フォールバックの番号付き平文で）必ずユーザーに選ばせる。選択肢は次の2つを提示する：

| 選択肢 | 対象 | 取得方法 |
|---|---|---|
| **現在の会話** | 今やり取りしているこのセッション | 自分の会話コンテキストをそのまま渡す |
| **今までの会話** | 過去に蓄積されたセッションログ全体 | `.claude/plugin-workspace/sessions/<id>/` を読む（skill-grower のフックがある場合のみ存在） |

「今までの会話」を選ばれたのにログのディレクトリが無いときは、その旨を伝え、`AskUserQuestion` で現在の会話に切り替えるか確認する。無理にログを探し回らない。

### Step 2: サブエージェントでナレッジ候補を情報収集する

候補の洗い出し（会話の走査・知見の抽出・整理）は **サブエージェントに行わせる**。本体のコンテキストを汚さず、観点ごとに掘り下げられるため。

- 「今までの会話」が対象で**セッションが複数ある／観点が複数ある**ときは、セッション単位・観点単位で **サブエージェントを並行起動** する。
- 各サブエージェントには「拾う対象・拾わない対象」と「候補の出力フォーマット」を渡し、構造化された候補リストを返させる。
- 本体はサブエージェントの結果を統合し、粒度を整える。**既存 issue との重複は気にしない**（被ってよい。目的は知見を取りこぼさず集めることなので、重複排除より網羅を優先する）。

サブエージェントに渡す抽出基準：

> **大前提（最優先）**: dev-knowledge の目的は「**プログラミングで得た技術的知見・遭遇した問題と解決策**」を貯めること。拾うべきは **他のプロジェクト・他の人にも転用できる技術ナレッジ**（言語/フレームワーク/API の挙動、アルゴリズム・データ構造、設計手法、環境・バージョン固有の挙動など）。**「このセッションで使っていたスキル/ツール/ワークフローがどう動くか」というメタな実況そのものは拾わない。** その作業の中に転用可能な技術的知見があれば、**スキル固有の手順説明ではなく、汎用化した技術ナレッジに言い換えて**抽出する（例: 「○○スキルは Step2 で JSON を編集する」ではなく「複数箇所を Markdown に反映するときは anchor を昇順処理すると位置ズレを防げる」）。

**拾う対象**（技術ナレッジ優先。dev-knowledge は「雑多に蓄積」が方針なので、技術的なら迷ったら拾う）
- 遭遇した問題と解決 — エラー・詰まり → 原因 → どう直したか、が揃っているもの
- 非自明な挙動・ハマりどころ — 「実はこうなっている」「ドキュメントと違った」
- 設計判断・トレードオフ — なぜその方式を選んだか、却下した案（**汎用化できる設計知見**として）
- 再利用できる手順・コマンド・スニペット — 次も同じ作業をしそうなもの
- 環境・バージョン固有の知見 — 「このバージョンだとこうなる」「このツールにはこの機能が無い」

**拾わない対象**
- 一度きりで再現条件の無い雑談
- **特定のスキル/ツール/ワークフローの使い方・内部手順の紹介そのもの**（「このスキルは〜する」「ワークフローは Step1→2→3」というメタな説明）。転用可能な技術的知見が中にあるときだけ、それを汎用化して拾う。
- 機密情報（API キー・トークン・パスワード・メールアドレス等）。混入していたらマスクする

各候補に付ける情報：**1行タイトル / 問題・背景 / 原因 or 経緯 / 解決 or 現状 / 関連タグ案 / ナレッジとして情報が十分か不足か**。

### Step 3: 候補リストを提示し、どれを起票するか AskUserQuestion で決めさせる

統合した候補を一覧で見せ、**どれを起票するかをユーザーに選ばせる**。`AskUserQuestion` が使える環境では候補を選択肢として提示し、複数選べるよう `multiSelect` を使う。**`AskUserQuestion` が無い環境では、候補を `1.` `2.` … と番号付きで提示し「起票する番号を答えてください（複数可・0で起票なし）」と明示してユーザーの返答を待つ**（フォールバックの詳細は冒頭の注記を参照）。いずれの環境でも、候補だけ出して止まったり、Claude が勝手に全部起票したりしない（何を残すかの判断はユーザーのもの）。各候補に、推奨する進め方ラベルを添える。

提示フォーマット：

```
## ナレッジ候補（現在の会話から N 件）

1. [推奨: knowledge:direct] <1行タイトル>
   - 問題/背景: ...
   - 解決/現状: ...（解決済み・情報十分）
   - tags 案: [react, hooks]

2. [推奨: knowledge:pr] <1行タイトル>
   - 問題/背景: ...
   - 現状: 解決の裏取りが無い・情報不足
   - tags 案: [rust, lifetime]
```

**進め方ラベルの判定基準**（最終決定はユーザー）：

- **`knowledge:direct`** — ナレッジとして **情報が十分**。会話の中で結論・裏取りが揃っていて、そのまま md 化できる。
- **`knowledge:pr`** — ナレッジとして **情報が不足**。裏取り・調査が要る、未解決、出典が曖昧。**情報不足の候補はこちらにする**。PR 側で調査して内容を補う前提なので、迷ったら `knowledge:pr`。

### Step 4: 選ばれた候補を 1 件ずつ起票する

ユーザーが選んだ各候補は **1 トピック = 1 issue** で `dio0550/dev-knowledge` に起票する。起票の手順・ラベル運用・`gh` が使えない環境での代替（GitHub MCP）はすべて `knowledge-issue` スキルに従う。そちらの本文テンプレートを各候補に当てはめる。

- 情報が十分（`knowledge:direct`）で内容が揃っているものは、先に `knowledge-article` で記事化してから「A. 記事を登録」で起票してもよい。
- 情報不足（`knowledge:pr`）のものは「B. 開発中の疑問・設計メモを起票する」テンプレートで、調べたいこと・仮説を残して起票する。
- 複数候補を 1 つの issue にまとめない。横断はタグで担保する方針なので、粒度は小さく保つ。

### Step 5: 起票結果を報告する

起票した issue 番号・URL・付けたラベルを一覧で報告する。ユーザーが見送ったものも明記して、取りこぼしが分かるようにする。

## 関連スキル

| スキル | 役割 | このスキルとの関係 |
|---|---|---|
| `knowledge-issue` | 1 件を dev-knowledge の issue に起票 | Step 4 の起票メカニクスを委譲する先 |
| `knowledge-article` | 知見を frontmatter 付き md 記事にする | 情報十分な候補を記事化してから起票する場合に使う |

## 前提条件

- 起票先は **`dio0550/dev-knowledge`**。進め方ラベル（`knowledge:pr` / `knowledge:direct`）が用意済みであること（未用意なら `knowledge-issue` の「ラベルの用意」を先に実行）。
- `gh`（GitHub CLI）が認証済みであること。使えない環境では GitHub MCP ツールで代替する。
- 「今までの会話」を対象にする場合のみ `.claude/plugin-workspace/sessions/` が存在すること（無ければ現在の会話に切り替える）。
