# 会話からナレッジ候補を棚卸しして issue にまとめて起票する

会話の中で「これは後で再利用できる」知見は、その場で記録しないと流れて消える。このスキルは、セッションを振り返って **ナレッジ候補を機械的に拾い上げ → 選別 → まとめて issue 化** する受け口。

`knowledge-article`（1記事を書く）/ `knowledge-issue`（1件を起票する）に対し、このスキルは **複数候補を一度に洗い出して仕分ける** 役を担う。1件ずつの起票そのものは `knowledge-issue` のルールに従う。

---

## ワークフロー

進捗チェックリストをコピーして進める：

```
- [ ] Step 1: データソースを決める
- [ ] Step 2: ナレッジ候補を抽出する
- [ ] Step 3: 既存 issue・記事との重複を除く
- [ ] Step 4: 候補リストを提示してユーザーに選別してもらう
- [ ] Step 5: 選ばれた候補を 1 件ずつ起票する
- [ ] Step 6: 起票結果を報告する
```

### Step 1: データソースを決める

| ソース | いつ使うか | 取得方法 |
|---|---|---|
| **今の会話**（既定） | 「この会話から」「今のセッションで」など | 自分の会話コンテキストをそのまま読む |
| **過去のセッションログ** | 「前のセッションも」「ログから」など | `.claude/plugin-workspace/sessions/<id>/` を読む（skill-grower のフックがある場合のみ存在） |

過去ログを求められたのにディレクトリが無いときは、その旨を伝えて今の会話だけで続行する。無理にログを探し回らない。

### Step 2: ナレッジ候補を抽出する

会話を走査し、以下に当てはまる断片を候補として拾う。dev-knowledge は「雑多に蓄積」が方針なので、**迷ったら拾う**（捨てるのは Step 4 でユーザーがやる）。

拾う対象：

- **遭遇した問題と解決** — エラー・詰まり → 原因 → どう直したか、が揃っているもの
- **非自明な挙動・ハマりどころ** — 「実はこうなっている」「ドキュメントと違った」など
- **設計判断・トレードオフ** — なぜその方式を選んだか、却下した案
- **再利用できる手順・コマンド・スニペット** — 次も同じ作業をしそうなもの
- **環境・バージョン固有の知見** — 「このバージョンだとこうなる」

拾わない対象：

- 一度きりで再現条件の無い雑談、結論の出ていない未解決の愚痴
- 機密情報（API キー・トークン・パスワード・メールアドレス等）。候補本文に混入していたらマスクする

各候補について、起票しやすいよう次をメモしておく：**1行タイトル / 問題・背景 / 原因 or 経緯 / 解決 or 現状 / 関連タグ案 / 解決済みか未解決か**。

### Step 3: 既存 issue・記事との重複を除く

同じ知見を二重起票しないため、既存とつき合わせる。

- 既存 issue: `gh issue list --repo dio0550/dev-knowledge --state all`（`gh` が無ければ `mcp__github__list_issues`）
- 既存記事: dev-knowledge の `docs/` 配下のタイトル・tags

タイトルやタグが近い候補は、新規にせず「既存 #N に追記」を提案する。判断に迷うものは候補に残し、Step 4 でユーザーに委ねる。

### Step 4: 候補リストを提示してユーザーに選別してもらう

抽出した候補を一覧で見せ、選んでもらう。**勝手に全部起票しない**（ノイズ起票はナレッジベースを汚すため）。各候補に、推奨する進め方ラベルも添える。

提示フォーマット：

```
## ナレッジ候補（今の会話から N 件）

1. [推奨: knowledge:direct] <1行タイトル>
   - 問題/背景: ...
   - 解決/現状: ...（解決済み）
   - tags 案: [react, hooks]

2. [推奨: knowledge:pr] <1行タイトル>
   - 問題/背景: ...
   - 現状: 未解決・要調査
   - tags 案: [rust, lifetime]

※ 重複候補: 3 は既存 #12 と近い → 追記を提案
```

進め方ラベルの推奨基準（最終決定はユーザー）：

- **`knowledge:direct`** — 会話の中で結論が出ている軽い知見。すぐ md 化できる
- **`knowledge:pr`** — 裏取り・調査が要る、未解決、レビューしたい。迷ったらこちら

### Step 5: 選ばれた候補を 1 件ずつ起票する

選別後の各候補は **1 トピック = 1 issue** で `dio0550/dev-knowledge` に起票する。起票の手順・ラベル運用・`gh` が使えない環境での代替（GitHub MCP）はすべて `knowledge-issue` スキルに従う。そちらの「B. 開発中の疑問・設計メモを起票する」本文テンプレートを各候補に当てはめる。

- 解決済みで内容が揃っているものは、先に `knowledge-article` で記事化してから「A. 記事を登録」で起票してもよい。
- 複数候補を 1 つの issue にまとめない。横断はタグで担保する方針なので、粒度は小さく保つ。

### Step 6: 起票結果を報告する

起票した issue 番号・URL・付けたラベルを一覧で報告する。重複でスキップしたもの、ユーザーが見送ったものも明記して、取りこぼしが分かるようにする。

## 関連スキル

| スキル | 役割 | このスキルとの関係 |
|---|---|---|
| `knowledge-issue` | 1 件を dev-knowledge の issue に起票 | Step 5 の起票メカニクスを委譲する先 |
| `knowledge-article` | 知見を frontmatter 付き md 記事にする | 解決済み候補を記事化してから起票する場合に使う |

## 前提条件

- 起票先は **`dio0550/dev-knowledge`**。進め方ラベル（`knowledge:pr` / `knowledge:direct`）が用意済みであること（未用意なら `knowledge-issue` の「ラベルの用意」を先に実行）。
- `gh`（GitHub CLI）が認証済みであること。使えない環境では GitHub MCP ツールで代替する。
- 過去ログを対象にする場合のみ `.claude/plugin-workspace/sessions/` が存在すること（無ければ今の会話で続行）。
